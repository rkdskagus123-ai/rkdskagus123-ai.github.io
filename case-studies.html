대표 프로젝트 케이스 스터디

공개 가능한 범위에서 문제, 구현 방향, 검증 흔적을 정리했습니다. 사내 URL, 실데이터, 민감 정보는 제외하고 재현 가능한 구조와 판단 기준을 중심으로 썼습니다.

01

Redmine/GitLab 주간보고 자동화

Problem

매주 개인 업무 내용을 Redmine wiki 형식으로 정리하고 GitLab 커밋 이력을 별도로 확인해야 했습니다. 수동 작성은 날짜, 상태, 진행률 누락이 생기기 쉬웠고, wiki 게시 후 수정 비용도 있었습니다.

Approach

보고서 원본은 Markdown outline으로 유지하고, Python CLI가 wiki 본문으로 변환하도록 했습니다. 게시 전 validation 단계에서 상태, 날짜, 진행률을 확인하고 API key는 환경변수로만 주입하도록 분리했습니다.

Evidence

python redmine_weekly_report.py --validate
확인 질문 없음: 상태, 진행률, 날짜가 모두 채워져 있습니다.

Public-safe Proof

샘플 outline, 샘플 output, validation command, 민감 정보 제거 스냅샷으로 실제 사내 정보 없이도 자동화 흐름을 확인할 수 있게 정리했습니다.

02

Exam Maker / Exam AI Analyzer

Problem

AI로 시험지 초안을 만들 수는 있지만, 중복 문항, 정답 분포, 난이도, DOCX 출력 형식은 별도의 검증 흐름이 필요했습니다. 실제 학생/교재 데이터는 테스트에 그대로 쓸 수 없었습니다.

Approach

Codex CLI 기반 생성 workflow를 preview, edit, JSON 검증, DOCX 출력 단계로 나눴습니다. Analyzer는 FastAPI로 분리하고 OCR provider, report API, pattern summary API를 경계로 잡았습니다.

Evidence

scripts/e2e-smoke.mjs
scripts/analyzer-pattern-summary.test.mjs
scripts/analyzer-dry-run-smoke.mjs

Public-safe Proof

synthetic 지문, validation warning, DOCX 출력 샘플, API smoke 결과를 기준으로 실제 학생/교재 데이터 없이도 검증 흐름을 설명합니다.

03

Smart Baby

Problem

영아 기록, 가족 공유, 사진/영상 분류는 민감 정보와 권한 문제가 함께 따라옵니다. AI 분석 결과를 확정 판단처럼 보이게 만들면 사용자 신뢰와 안전에 문제가 생깁니다.

Approach

Flutter/Riverpod/Drift 기반으로 care event, health record, family sharing, album classification을 도메인 모델로 분리했습니다. AI 결과는 candidate와 review state를 따로 두어 불확실성을 표현했습니다.

Evidence

548 commits
84 test or fixture related files
WebRTC signaling experiment included

Public-safe Proof

실제 아이 사진이나 가족 정보 대신 demo data, domain model, review state를 기준으로 민감 데이터 처리 방식을 보여줍니다.

04

Jungdon / My Refrigerator

Problem

냉장고와 수납장의 물건은 위치와 유통기한을 기억하기 어렵고, 입력이 번거로우면 앱을 계속 쓰기 어렵습니다. 로컬 우선 사용성과 향후 동기화 확장성 사이의 균형도 필요했습니다.

Approach

Flutter/Riverpod/Drift 기반 local-first 구조를 잡고 물품 CRUD, 위치 트리, 검색, 유통기한 목록, 바코드 스캔, Open Food Facts 제품명 조회 흐름을 구현했습니다.

Evidence

49 commits
78 test/smoke/spec related files
barcode scanner + Open Food Facts flow

Public-safe Proof

데모 재고 데이터, local DB 구조, 바코드/제품명 조회 흐름을 통해 생활형 앱의 입력 부담과 검색 흐름을 설명합니다.

Publishing Rule

공개 포트폴리오에는 실제 고객명, 사내 URL, 장비 고유 정보, 실데이터를 넣지 않습니다. 대신 구조, 흐름, 검증 방식, 익명화된 샘플 결과를 보여주는 쪽이 더 안전하고 오래 갑니다.
